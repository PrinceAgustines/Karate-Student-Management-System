# Attendance Module - Implementation Summary

## Overview
The Karate Student Management System includes a comprehensive attendance tracking module with facial recognition capabilities, session management, and detailed attendance logs. The system spans both frontend (React/TypeScript) and backend (Django) implementations.

---

## 1. Frontend File Structure

### Location: `frontend/src/app/components/pages/attendance/`

#### Files:
1. **AttendanceLogs.tsx**
   - Purpose: Displays historical attendance records and logs
   - Features:
     - Search functionality (by student name or session)
     - Status filtering (present/absent/all)
     - Date filtering (today/week/month/all)
     - Statistics dashboard (total present, total absent, avg recognition confidence)
     - Exportable CSV functionality
     - Displays recognition confidence percentage
     - Shows attendance method (Camera vs Manual)

2. **AttendanceTracker.tsx**
   - Purpose: Real-time attendance marking for active sessions
   - Features:
     - Session selection dropdown
     - Student list with attendance status toggle
     - Session information display (date, time, venue, instructor, enrolled count)
     - Camera integration placeholder for facial recognition
     - Simulated facial recognition with confidence scores (85-100%)
     - Statistics display (total students, present, absent)
     - Search students by name
     - Manual attendance toggle (Present/Absent/Unmarked)

3. **DojoSchedule.tsx**
   - Purpose: Unified schedule page for creating, viewing, and managing sessions
   - Features:
     - Weekly calendar view with Sunday-first ordering
     - One place for create/edit/delete session workflows
     - Minimal session form fields: type, date, start/end time, venue
     - Session list and schedule cards display venue and instructor status
     - Read-only schedule for non-admin users
     - No separate Session Management page exists in the current implementation

## 2. Backend Models

### Location: `backend/students/models.py`

#### Core Attendance Models:

1. **Session Model**
   ```
   - session_id (AutoField, Primary Key)
   - date (DateField)
   - start_time (TimeField)
   - end_time (TimeField)
   - venue (CharField)
   - session_type (CharField with choices)
     * 'regular'
     * 'seminar'
     * 'competition prep'
   - instructor (ForeignKey → Student)
   
   Related: Has many Attendance records via attendances
   ```

2. **Attendance Model**
   ```
   - attendance_id (AutoField, Primary Key)
   - student (ForeignKey → Student) [CASCADE]
   - session (ForeignKey → Session) [CASCADE]
   - date (DateField)
   - time_in (TimeField, nullable)
   - time_out (TimeField, nullable)
   - recognition_confidence (DecimalField, 5 digits, 2 decimals, nullable)
   
   Allows: Tracking check-in/check-out times
           Storing facial recognition confidence scores
           Supporting both camera and manual tracking
   ```

#### Related Models (Supporting Data):
- **Student**: Referenced by both Session (as instructor) and Attendance (as attendee)
- **FaceData**: Stores facial encodings for facial recognition
  - face_encoding (TextField)
  - image_path (CharField)
  - face_angle (CharField: Front/Left/Right)
  - date_registered (DateField)

## 3. API Endpoints

### Location: `backend/students/urls.py` and `backend/students/views.py`

#### ViewSet Routes (Auto-generated by Django REST Framework):

1. **Attendance Endpoints**
   ```
   GET    /api/students/attendances/          - List all attendance records
   POST   /api/students/attendances/          - Create new attendance record
   GET    /api/students/attendances/{id}/     - Retrieve specific attendance record
   PUT    /api/students/attendances/{id}/     - Update attendance record (full)
   PATCH  /api/students/attendances/{id}/     - Partial update
   DELETE /api/students/attendances/{id}/     - Delete attendance record
   ```

2. **Session Endpoints**
   ```
   GET    /api/students/sessions/             - List all sessions
   POST   /api/students/sessions/             - Create new session
   GET    /api/students/sessions/{id}/        - Retrieve specific session
   PUT    /api/students/sessions/{id}/        - Update session (full)
   PATCH  /api/students/sessions/{id}/        - Partial update
   DELETE /api/students/sessions/{id}/        - Delete session
   ```

#### Permission Requirements:
- All attendance and session endpoints require authentication (`permissions.IsAuthenticated`)
- Bearer token required in Authorization header

## 5. Data Tracking Methods

### Attendance Tracking Mechanism:

1. **Facial Recognition (Camera-based)**
   - Stores `recognition_confidence` score (0-100%)
   - Records `time_in` when face is recognized
   - Tracks confidence level for audit purposes
   - Placeholder UI with "Start Facial Recognition" button
   - Uses `FaceData` model for storing facial encodings

2. **Manual Tracking (UI-based)**
   - Instructors can manually mark attendance
   - Toggle students Present/Absent/Unmarked
   - No confidence score recorded
   - System displays "Manual" as method when no confidence score exists
   - Simulated facial recognition can assign confidence (85-100%)

3. **Time Tracking**
   - `time_in`: Check-in time (first facial detection or manual mark)
   - `time_out`: Check-out time (optional, for session end)
   - Both fields optional to support partial attendance tracking

4. **Session Association**
   - All attendance is linked to a specific session
   - Session defines date, time, venue, instructor, and session type
   - Enrolled count automatically calculated from attendance records

## 6. Database Schema

### Key Tables:

**students_session**
```
- session_id (PRIMARY KEY)
- date
- start_time
- end_time
- venue
- session_type
- instructor_id (FOREIGN KEY → students_student)
```

**students_attendance**
```
- attendance_id (PRIMARY KEY)
- student_id (FOREIGN KEY → students_student)
- session_id (FOREIGN KEY → students_session)
- date
- time_in
- time_out
- recognition_confidence
```

**students_facedata**
```
- id (PRIMARY KEY)
- student_id (FOREIGN KEY → students_student)
- face_encoding
- image_path
- face_angle
- date_registered
```

### Relationships:
```
Student ──<1─── Session (via instructor_id)
Student ──<*─── Attendance (via student_id)
Session ──<*─── Attendance (via session_id)
Student ──<*─── FaceData (via student_id)
```

## 11. Integration Notes

- All frontend API calls use `buildHeaders(true)` for authentication
- API base URL from environment: `VITE_API_BASE` (defaults to `http://127.0.0.1:8000`)
- Attendance data flows: UI captures input → API POST/PATCH → Django validates → Database stores
- Sessions are required entities; attendance records must be linked to a session
- Confidence scores range 0-100% for facial recognition validation
